# OAT Dogfood Workflow Implementation - Design v2

**Date:** 2026-01-27
**Status:** Approved (post-Codex feedback)
**Purpose:** Revised design for implementing the OAT workflow system by dogfooding it to build itself

**Changes from v1:**
- Workflow ordering: discovery → spec → design → plan → implement
- Projects stay in `.agent/projects/` (not `.oat/projects/`)
- Drop SCREAMING_SNAKE naming (use regular case + frontmatter)
- Phase-based HiL gates (not task-based)
- Add `oat-design` skill and `oat-progress` router skill
- Per-phase logs as mode (default single file, parallel when needed)
- Enhanced staleness detection (age + scoped changes)

---

## Overview & Scope

We're implementing a two-phase system to dogfood the OAT workflow:

### Phase 1: Knowledge Generation

- Adapt GSD's map-codebase parallel mapper pattern
- Generate `.oat/knowledge/project-index.md` (SuperClaude-style overview)
- Generate `.oat/knowledge/codebase/` with 7 files: architecture.md, stack.md, structure.md, integrations.md, testing.md, conventions.md, concerns.md
- Enhanced staleness detection (age + scoped file/line changes)
- Skill: `oat-index` orchestrates 5 agents:
  - 1 for project-index.md (sequential, SuperClaude pattern)
  - 4 parallel GSD mappers for codebase files

### Phase 2: Workflow Skills

- Five core skills: `oat-discovery`, `oat-spec`, `oat-design`, `oat-plan`, `oat-implement`
- Plus router skill: `oat-progress`
- Each skill updates `state.md` inline
- Full mode only (thorough questioning, detailed artifacts)
- Single `implementation.md` file (switch to per-phase logs when parallel execution needed)
- Atomic commits per task with conventional commit format

### Deferred for Later

1. Parallel worktree support (per-phase logs automatic when parallel declared)
2. Quick mode toggle
3. Standalone `oat-state-update` skill
4. CLI template rendering service
5. Memory system (`.oat/memory/`)
6. `oat-reconcile` skill (for parallel phase execution)

### Skill Coordination

Skills pass data via **explicit file reads with conventional paths:**

```

oat-plan       → reads design.md + spec.md + quick-spec.md, writes plan.md + state.md
oat-implement  → reads state.md → plan.md → quick-spec.md, updates all three
oat-progress   → reads state.md + all artifacts, reports status + recommends next skill
```

## Workflow Phases

```
discovery (why) → spec (what) → design (how) → plan (steps) → implement (do)
```

### Artifact Relationships

**discovery.md** = Conversation record (iteratively updated during `oat-discovery`)
- Questions asked and answers received
- Options considered with tradeoffs
- Decisions made and rationale
- The thinking process and exploration
- Raw, chronological conversation log
- Updated continuously during discovery conversation

**spec.md** = Requirements specification
- Must/should/could requirements
- Acceptance criteria
- Explicit scope boundaries (in/out)
- Success metrics
- **High-Level Design (Proposed)** section (seeds design.md in Full mode)

**quick-spec.md** = Agent-optimized requirements
- 2-3 paragraph summary of MVP scope
- Key constraints
- Link to full spec.md

**design.md** = Architecture & approach (reviewable checkpoint)
- Architecture decisions (directory structure, data models, patterns)
- Technical approach (chosen from options explored in discovery)
- Integration points with existing codebase
- Trade-offs analysis (refined from spec's proposed approach)
- Clean, structured document for review
- **Approval gate before planning**

**plan.md** = Execution tasks
- Phased tasks with verification steps
- Specific file changes
- HiL gates marked by phase number

**implementation.md** = Execution log
- Work done, decisions, ADRs
- Project-specific knowledge (conventions, context)
- Should be read before picking up a phase's work

## Data Model & Frontmatter

### Knowledge Artifacts (Generated)

**project-index.md and codebase/*.md**
```yaml
---
oat_generated: true
oat_generated_at: 2026-01-27
oat_default_branch: origin/main
oat_source_main_merge_base_sha: abc123def456
oat_source_paths:
  - "apps/**"
  - "packages/**"
  - "src/**"
oat_warning: "GENERATED FILE - Do not edit manually. Regenerate with /oat:index"
---
```

**Staleness check (enhanced):**
```
1. Age: (now - oat_generated_at) > 7 days → warn

2. Git-based change detection (if available):
   - git diff --name-only <sha>..HEAD | match paths → count changed files
   - git diff --shortstat <sha>..HEAD → parse +/- lines
   - If file count > threshold OR line delta > threshold → warn

3. Fallbacks for non-git or problematic git state:
   - Non-git directory → age-only, warn "git-based staleness unavailable"
   - Missing origin/main → use HEAD as anchor, warn "no upstream reference"
   - Shallow clone → age-only, warn "shallow clone limits diff accuracy"
   - Detached HEAD → age-only, warn "detached HEAD, no stable diff anchor"

4. Decision logic:
   - If age high OR (git changes high when available) → warn
   - v1: warn only, no hard block
   - Future: strict mode can block on stale knowledge
```

### Workflow Artifacts (Working Docs)

**discovery.md, spec.md, quick-spec.md, design.md, plan.md**
```yaml
---
oat_status: draft # draft|in_review|approved|superseded
oat_ready_for: oat-spec # oat-spec|oat-design|oat-plan|oat-implement|null
oat_blockers: [] # reference IDs; empty means unblocked
oat_last_updated: 2026-01-27
---
```

**Blocker format (reference IDs in frontmatter, details in body):**
```yaml
---
oat_blockers: [B1, B2]  # empty array = unblocked
---

## Blockers

**B1**: Need to decide between JWT vs session-based auth
- Options: JWT (stateless) vs Redis sessions (stateful)
- Impact: Affects horizontal scaling strategy
- Decision needed by: Phase 2 Task 5

**B2**: Waiting on API key from third-party service
- Service: Stripe payment integration
- Requested: 2026-01-26
- Blocking: Phase 3 Task 2
```

**plan.md (additional fields)**
```yaml
---
oat_status: approved
oat_ready_for: oat-implement
oat_blockers: []
oat_last_updated: 2026-01-27
oat_hil_phases: [1, 4, 5, 7] # phase numbers requiring HiL gates
oat_parallel_execution: false # true = use per-phase logs + UAT files
---
```

**Parallel execution requirements:**
When `oat_parallel_execution: true`:
- **Required:** Per-phase implementation logs (`implementation-phase-<NN>.md`)
- **Required:** Per-phase UAT files for HiL phases (`uat-phase-<NN>.md`)
- **Required:** `implementation.md` becomes index only (not directly edited)
- **Benefit:** Avoids merge conflicts across worktrees
- **Setup:** Each worktree works on independent phase(s)
- **Reconciliation:** Use `oat-reconcile` skill (future) to merge phase results

**state.md (routing + recovery)**
```yaml
---
oat_current_task: p02-t03  # Phase derived from prefix (p02 = Phase 2)
oat_last_commit: abc123
oat_last_updated: 2026-01-27
oat_blockers: []
---

## Current Position
Phase 2: Implementation (derived from p02-*)
Task: p02-t03 - Implement user authentication

## Next Actions
1. Complete auth middleware
2. Add tests for auth flow
3. Update API documentation

## Blockers
(none)
```

**Phase numbering convention:**
- Phase N is identified by task prefix `pNN`
- No separate `oat_current_phase` field (derived from `oat_current_task`)
- Example: `p02-t03` → Phase 2, Task 3
- Prevents drift between phase number and task prefix

**implementation.md**
```yaml
---
oat_status: in_progress
oat_last_updated: 2026-01-27
---
```

### Implementation Logs: Mode-Based Pattern

**Default (serial execution):** Single `implementation.md` with phase sections
```markdown
## Phase 1: Knowledge Generation

### Session 2026-01-27
- Generated project-index.md
- Spawned 4 parallel mappers
- All 8 files created successfully

### ADRs
**ADR-001**: Use GSD mapper pattern for codebase analysis
- Rationale: Proven pattern, parallel execution, good templates
- Alternatives: SuperClaude-only (too shallow), manual (too slow)

## Phase 2: Authentication

### Session 2026-01-27
[Agent writes here - do not edit other phase sections]
```

**Parallel mode (when plan.md declares parallel execution):**
- `implementation.md` becomes index
- Each phase writes to `implementation-phase-<NN>.md`
- Automatic when `oat_parallel_execution: true` in plan.md

## Skill Specifications

### Skill: `oat-index`

**Purpose:** Generate repo-wide knowledge base (project-index.md + 7 codebase files)

**Inputs:**
- Current codebase
- Optional: existing `.oat/knowledge/` for refresh check

**Process:**
1. Check if `.oat/knowledge/` exists
   - If exists: offer Refresh/Skip
2. Generate project-index.md (sequential, SuperClaude pattern)
   - Quick scan of repo structure
   - High-level overview (what/why/who)
   - Key technologies and patterns
3. Spawn 4 parallel `oat-codebase-mapper` agents with `run_in_background=true`:
   - Agent 1 (tech): stack.md + integrations.md
   - Agent 2 (arch): architecture.md + structure.md
   - Agent 3 (quality): conventions.md + testing.md
   - Agent 4 (concerns): concerns.md
4. Collect confirmations (read agent output files for file paths + line counts)
5. Verify output (`ls -la`, `wc -l`)
6. Commit with `docs: generate knowledge base`
7. Staleness warning setup (age + scoped changes)

**Outputs:**
- `.oat/knowledge/project-index.md`
- `.oat/knowledge/codebase/*.md` (7 files)
- Git commit

**Exit criteria:**
- All 8 files exist and non-empty (>20 lines each)
- Frontmatter includes generation metadata
- User informed of next step: `/oat:discovery`

### Skill: `oat-discovery`

**Purpose:** Gather requirements, constraints, open questions through interactive interview

**Inputs:**
- `.oat/knowledge/project-index.md` (and codebase map if exists)
- Prior research/notes (optional)
- User responses to questions

**Process:**
1. Read knowledge base for context
2. Ask questions one at a time:
   - What problem are we solving?
   - Who are the users?
   - What are the constraints?
   - What does success look like?
   - What are the open questions?
3. Propose 2-3 approaches with tradeoffs
4. Synthesize into discovery.md (using template)
5. Update state.md with next action

**Template structure (discovery.md):**
- Problem Statement
- Target Users
- Constraints
- Success Criteria
- Approaches Considered (2-3 with tradeoffs)
- Open Questions / Blockers
- Recommended Approach

**Outputs:**
- `.agent/projects/<name>/discovery.md`
- `.agent/projects/<name>/state.md` (created/updated)

**Exit criteria:**
- `oat_status: approved` in discovery.md frontmatter
- `oat_ready_for: oat-spec`
- `oat_blockers: []` (or explicit blockers listed with IDs)

---

### Skill: `oat-spec`

**Purpose:** Transform discovery into stable requirements specification

**Inputs:**
- `.agent/projects/<name>/discovery.md`
- User clarifications on scope/priority

**Process:**
1. Read discovery.md
2. Structure requirements into:
   - Must have (MVP scope)
   - Should have (important but deferrable)
   - Could have (nice-to-have)
   - Won't have / Out of scope (explicit non-goals)
3. Define acceptance criteria for must-haves
4. List risks and unknowns
5. Add **High-Level Design (Proposed)** section
6. Generate both full spec.md and concise quick-spec.md
7. Update state.md

**Template structure (spec.md):**
- Overview (problem, solution, users)
- Requirements
  - Must Have (with requirement IDs: REQ-001, REQ-002...)
  - Should Have
  - Could Have
- Non-Goals (explicit out-of-scope)
- Success Criteria / Acceptance Checks
- **High-Level Design (Proposed)** ← Seeds design.md in Full mode
  - Goals / constraints (design-relevant)
  - Proposed approach (1-2 paragraphs)
  - Key decisions (bullets)
  - Alternatives considered (2-3 bullets + why rejected)
  - Open questions / risks
  - Impacted areas (high-level modules/files)
- Risks & Unknowns
- Dependencies

**Template structure (quick-spec.md):**
- 2-3 paragraph summary of MVP scope
- Key constraints
- Link to full spec.md for details

**Usage pattern:**
- **spec.md**: Complete requirements, read when need specific requirement details
- **quick-spec.md**: MVP scope summary, read when starting work, context refresh
- **oat-implement**: ALWAYS loads quick-spec.md first (cheap context), only loads spec.md if task references specific REQ-ID

**Outputs:**
- `.agent/projects/<name>/spec.md`
- `.agent/projects/<name>/quick-spec.md`
- `.agent/projects/<name>/state.md` (updated)

**Exit criteria:**
- `oat_status: approved` in both spec.md and quick-spec.md
- `oat_ready_for: oat-design`
- Must/Should/Could and non-goals explicit
- Acceptance checks defined
- High-Level Design section complete

### Skill: `oat-design`

**Purpose:** Create reviewable architecture document from spec's proposed approach

**Inputs:**
- `.agent/projects/<name>/spec.md` (especially High-Level Design section)
- `.agent/projects/<name>/discovery.md` (for options context)
- Current codebase (for integration points)

**Process:**
1. Read spec.md High-Level Design section
2. Read discovery.md for explored options
3. Scan codebase for integration points
4. Expand proposed approach into full design:
   - Architecture decisions (directory structure, data models, patterns)
   - Technical approach (libraries, frameworks, algorithms)
   - Integration points with existing codebase
   - Trade-offs analysis (why this approach over alternatives)
   - Implementation considerations
5. Generate design.md (using template)
6. Update state.md

**Template structure (design.md):**
- Overview (problem recap, chosen approach)
- Architecture Decisions
  - Directory structure
  - Data models / schemas
  - Key patterns
- Technical Approach
  - Libraries / frameworks
  - Algorithms / strategies
  - Integration points
- Trade-offs Analysis
  - Why this approach
  - Alternatives considered (from discovery + spec)
  - Decision rationale
- Implementation Considerations
  - Migration path (if applicable)
  - Risk mitigation
  - Testing strategy
- Open Questions (if any remain)

**Outputs:**
- `.agent/projects/<name>/design.md`
- `.agent/projects/<name>/state.md` (updated)

**Exit criteria:**
- `oat_status: approved` in design.md frontmatter
- `oat_ready_for: oat-plan`
- **Design is the approval gate before planning**
- Architecture decisions documented
- Trade-offs explicit

**When design.md is required:**
- Always in Full mode (default for dogfood)
- Introduces/changes user-facing interface
- Cross-cutting architecture decisions
- Provider interoperability decisions
- New persistent data models or directory layouts

### Skill: `oat-plan`

**Purpose:** Turn approved design into executable implementation plan

**Inputs:**
- `.agent/projects/<name>/design.md` (must be approved)
- `.agent/projects/<name>/spec.md`
- `.agent/projects/<name>/quick-spec.md`
- Current codebase (for file path discovery)
- User input on HiL checkpoint placement

**Process:**
1. Verify design.md is approved (check `oat_status: approved`)
2. Read design + spec + quick-spec
3. Scan codebase for relevant patterns/conventions
4. Break work into phases (logical groupings from design)
5. Within each phase, define tiny tasks (2-5 minutes each)
6. For each task:
   - Task ID (p01-t01, p01-t02...)
   - Clear description
   - Exact file paths involved
   - Verification step (how to confirm it's done)
   - Link to spec requirement (REQ-XXX) if applicable
7. **Ask user where they want HiL checkpoints:**
   ```
   I've outlined X phases with Y total tasks.

   Where do you want Human-in-the-Loop checkpoints?

   Checkpoints pause implementation for your review/approval before continuing.

   Options:
   A) After each phase (conservative, max control)
   B) After major milestones only (I recommend: phases 2, 4, 7)
   C) Only at the end (fast, minimal interruption)
   D) Custom - you specify which phases

   What works for you?
   ```
8. Mark HiL phases in frontmatter based on user choice
9. Ask if parallel execution planned (sets oat_parallel_execution flag)
10. Update state.md

**Template structure (plan.md):**
```markdown
---
oat_hil_phases: [1, 4, 5, 7] # phase numbers requiring HiL gates
oat_parallel_execution: false # true = use per-phase logs
---

## Phase 1: <name>

**Goal:** <1-2 sentence phase goal>

**Tasks:**
- [ ] p01-t01: <description>
  - Files: `path/to/file.ts`
  - Verify: <test command or manual check>
  - Spec: REQ-001

- [ ] p01-t02: <description>
  - Files: `path/to/other.ts`
  - Verify: `pnpm test`
  - Spec: REQ-002

## Phase 2: <name>

**Goal:** <1-2 sentence phase goal>

**Tasks:**
- [ ] p02-t01: <description>
  ...
```

**Outputs:**
- `.agent/projects/<name>/plan.md`
- `.agent/projects/<name>/state.md` (updated)

**Exit criteria:**
- `oat_status: approved` in plan.md
- `oat_ready_for: oat-implement`
- Each task has file paths + verification
- HiL gates marked in frontmatter (based on user input)
- Parallel execution flag set if needed

### Skill: `oat-implement`

**Purpose:** Execute plan tasks in batch mode until hitting a checkpoint, with atomic commits and state recovery

**Inputs:**
- `.agent/projects/<name>/state.md` (ALWAYS read first)
- `.agent/projects/<name>/plan.md`
- `.agent/projects/<name>/quick-spec.md` (always loaded)
- `.agent/projects/<name>/spec.md` (loaded only if task references REQ-ID)
- `.agent/projects/<name>/implementation.md` (read for project context/ADRs before starting)

**Process:**

#### 1. State Recovery & Position Detection

```
1. Read state.md → get current_task (e.g., p02-t03) and last_commit
2. Read git log -1 → parse scope for task ID
3. Compare:
   - If commit task > state task → state is stale, update state.md first
   - If commit has no OAT pattern → prompt user to confirm position
   - If aligned → proceed
4. Determine current position in plan.md
```

#### 2. Batch Execution

```
1. Read implementation.md for project context (ADRs, conventions)
2. Load quick-spec.md for context
3. Starting from current_task, execute tasks sequentially:

   For each task:
   a. Execute the work (write code, run commands, etc.)
   b. Run verification step from plan
   c. Commit with pattern: feat(pXX-tYY): description
   d. Update plan.md checkbox with commit hash
   e. Check if current phase is in oat_hil_phases:
      - If YES → stop after phase completes, prompt user for approval
      - If NO → continue to next task
   f. Check for unplanned decision points → stop if found

4. Stop conditions:
   - Reached end of HiL phase (from oat_hil_phases)
   - Encountered unplanned decision point
   - Verification failed
   - All tasks complete
```

#### 3. Artifact Updates

```
After each task:
- plan.md: - [x] pXX-tYY: description (abc123)
- implementation.md: append session entry (or phase-specific log if parallel)
- state.md: update current_task, last_commit

Before stopping:
- state.md: update next actions, blockers if any
- implementation.md: update ADRs if architectural decisions made
```

**Commit Pattern:**
```bash
# Task-level work
feat(p02-t03): implement user authentication

- Added login endpoint
- Created JWT validation middleware

# Verification passed: pnpm test (all green)
```

**State Recovery Example:**
```
# Scenario: Process crashed after completing p02-t03 but before updating state

1. oat-implement starts → reads state.md (says p02-t02)
2. Reads git log -1 → sees feat(p02-t03): ...
3. Detects mismatch → "State shows p02-t02 but last commit is p02-t03"
4. Updates state.md to p02-t03, last_commit: abc123
5. Continues from p02-t04
```

**Human Commit Handling:**
```
# Scenario: Human made commits without OAT pattern

1. oat-implement starts → reads state.md (says p02-t03)
2. Reads git log -1 → "fix: typo in readme" (no pXX-tYY pattern)
3. Warns: "Last commit doesn't match OAT pattern. State may be out of sync."
4. Prompts: "Confirm current position: [p02-t03] or specify:"
5. User confirms or corrects
6. Updates state.md with confirmed position
7. Continues execution
```

**Outputs:**
- Code changes + git commits (one per task)
- `.agent/projects/<name>/plan.md` (checkboxes updated with commit hashes)
- `.agent/projects/<name>/implementation.md` (session log + ADRs appended)
- `.agent/projects/<name>/state.md` (position + blockers updated)

**Exit criteria:**
- Stopped at HiL phase boundary OR
- Stopped at unplanned decision point OR
- All tasks complete
- State accurately reflects position
- All completed tasks committed

### Skill: `oat-progress`

**Purpose:** Router skill that reports current state and recommends next action

**Inputs:**
- `.agent/projects/<name>/state.md`
- All workflow artifact frontmatter
- `.oat/knowledge/` (for staleness check)

**Process:**
1. **Check knowledge status FIRST (knowledge-first enforcement):**
   - Check if `.oat/knowledge/project-index.md` exists
   - If missing → recommend `/oat:index` immediately, block other recommendations
   - If exists → check staleness (age + git diff if available)
   - If stale → warn prominently, recommend `/oat:index` as first priority

2. Read state.md to determine current position
3. Check all artifact statuses (frontmatter)
4. Determine what's complete, in progress, blocked
5. Recommend next skill to run:
   - Priority 1: `/oat:index` if knowledge missing/stale
   - Priority 2: Next workflow skill based on state
6. Report clear next actions

**Knowledge-first routing:**
- Always check knowledge before recommending workflow skills
- Stale/missing knowledge = prominent warning + index recommendation
- Ensures "knowledge-first" is behaviorally enforced, not just suggested

**Output format:**
```
# Project Progress: <project-name>

## Current Status
Phase: Discovery
Status: In Progress
Last Updated: 2026-01-27

## Completed
✅ Knowledge Generation (.oat/knowledge/ generated 2 days ago)

## In Progress
🔄 Discovery - gathering requirements
   Status: draft
   Next: Complete discovery questions, then approve

## Blocked
(none)

## Next Action
Continue with /oat:discovery to complete requirements gathering.
Once discovery is approved, run /oat:spec to create requirements spec.

## Knowledge Status
⚠️ Warning: Knowledge base is 8 days old and 15 files changed since generation.
   Consider regenerating with /oat:index before proceeding.
```

## Commit Conventions

### OAT Commit Format

All commits follow conventional commits with OAT-specific scoping:

```
<type>(<scope>): <description>

<body>

<footer>
```

### Scope Patterns

**Task-level work (most common):**
```
feat(p02-t03): implement user authentication
docs(p01-t05): add API documentation
test(p03-t02): add integration tests
fix(p02-t08): handle edge case in validation
```

**Phase-level work (no specific task):**
```
feat(p02): complete authentication phase
docs(p02): update phase documentation
chore(p02): refactor phase structure
```

**Cross-phase work (exceptional):**
```
feat(px): update shared utilities
refactor(px): extract common patterns
chore(px): update dependencies
```

**Pre-phase work (no scope):**
```
feat: scaffold project structure
docs: generate knowledge base
chore: configure tooling
```

### Parsing Rules

**Regex for task extraction:** `\(p(\d+)-t(\d+)\)`
**Regex for phase extraction:** `\(p(\d+)\)(?!-)`
**Cross-phase:** `\(px\)`
**No scope:** Pre-phase or general work

### State/Plan Integration

**plan.md task format:**
```markdown
- [x] p02-t03: Implement user authentication (abc123)
      ↑ Task ID    ↑ Description              ↑ Commit hash
```

**state.md tracking:**
```yaml
---
oat_current_task: p02-t03
oat_last_commit: abc123def456
---
```

### Recovery Algorithm

```
1. Parse git log -1 for scope pattern
2. Extract task ID (pXX-tYY) if present
3. Compare to state.md current_task
4. If mismatch:
   - Commit ahead of state → update state
   - State ahead of commit → verify with user
   - No pattern in commit → prompt user confirmation
```
