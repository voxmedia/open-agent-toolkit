## Security Considerations

**Git Hooks Use node Instead of Bash for Portability:**
- Risk: `tools/git-hooks/manage-hooks.js` is a Node.js script, but git hooks conventionally run any interpreter. If Node.js is not in PATH or is misconfigured, hooks could silently fail or hang on some systems.
- Files: `tools/git-hooks/manage-hooks.js`, `.git/hooks/` (generated at runtime)
- Current mitigation: Script uses standard `node` invocation; assumes Node.js is available. Package.json enforces `node >=22.17.0`.
- Recommendations: 
  - Add a fallback bash wrapper or pre-hook check to verify Node.js availability.
  - Document git hook setup requirements and troubleshooting in `tools/git-hooks/README.md`.
  - Consider adding a pre-install hook to validate environment before setup completes.

**MCP Config Ignored But API Keys May Leak:**
- Risk: `.mcp.json` (Claude Code MCP configuration containing API keys) is gitignored, but if accidentally committed, it would expose credentials
- Files: `.gitignore` (line 32: `.mcp.json`), any `.mcp.json` accidentally created in repo root
- Current mitigation: `.mcp.json` is properly gitignored. No hardcoded secrets in source code.
- Recommendations:
  - Add a pre-commit hook (already in place via commitlint) to catch `.mcp.json` or `.env.*` additions.
  - Document that `.mcp.json` must never be committed in AGENTS.md or security guidelines.
  - Consider a `.mcp.json.example` template if MCP config is expected for local development.

**Process Exit Codes and Error Handling Vary:**
- Risk: Scripts use different exit codes (0, 1, 2) without clear documentation. Callers may not distinguish between recoverable errors (1) and environment errors (2).
- Files: `.oat/scripts/validate-oat-skills.ts` (exit 0, 1, 2), `.oat/scripts/new-oat-project.ts` (exit 0, 1), `.agent/scripts/new-agent-project.ts` (exit 0, 1)
- Current mitigation: Scripts provide error messages to stderr; callers can check exit code.
- Recommendations:
  - Standardize exit codes across scripts: define 0=success, 1=user error (bad args, validation failure), 2=system error (file I/O, missing deps)
  - Document exit code semantics in each script header or in a shared constants module.
  - Add error recovery guidance for non-zero exits in skill/CLI help text.

## Dependencies at Risk

**Turbo Build System Configuration Implicit:**
- Risk: Build dependencies and task dependencies are inferred by Turbo based on `workspace:*` and `^build` patterns. If dependencies drift, build order could become incorrect without obvious indication.
- Impact: Silent build failures or incorrect task ordering, especially after pnpm/Turbo updates
- Migration plan: 
  - Document build dependency rules in a `ARCHITECTURE.md` or `TURBOREPO.md` file
  - Add a build verification step to CI to catch out-of-order builds
  - Consider explicit `turbo.json` task dependency graph documentation

**TypeScript Strict Mode with Several Exceptions:**
- Risk: `strict: true` is enabled, but several rules are relaxed (`noNonNullAssertion: off`, `noExtraNonNullAssertion: off`). This could hide unsafe code.
- Impact: Type errors may not be caught until runtime, especially around null/undefined handling
- Migration plan:
  - Review and remove exceptions (`noNonNullAssertion: off`) if possible
  - Add runtime null checks where non-null assertions are used
  - Document why exceptions are necessary and revisit quarterly

**Node.js 22.17.0 Requirement May Limit Adoption:**
- Risk: Very recent Node.js requirement (`>=22.17.0`) means most CI/CD systems and production environments may not have it yet. Could block adoption or CI compatibility.
- Impact: Installation failures, CI pipeline setup friction, potential for version mismatches
- Migration plan:
  - Monitor Node.js LTS timeline and plan to support LTS versions (currently 20.x)
  - Document how to use `nvm` or `fnm` to manage Node.js versions
  - Consider relaxing requirement to `>=18.19.0` (current LTS) if there are no breaking dependency requirements for 22
