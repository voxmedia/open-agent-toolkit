## Summary
The plan is close to implementation-ready and generally follows the spec-driven OAT structure. The main gaps are that the remove flow does not actually cover agents despite the approved requirements, the plan adds an unapproved `oat tools info` command that upstream artifacts explicitly defer, and the agent-versioning task targets generated assets instead of the source-of-truth files that produce them.

### Important
- **`oat tools remove` is planned around skill-only logic, so agent removal is still uncovered** (`plan.md:561-615`)
- Issue: Discovery requires `remove` to cover both skills and agents (`discovery.md:83-84`), and the spec/design keep that contract for FR8 and the remove wrapper (`spec.md:109-117`, `design.md:194-212`). But p03-t02 only delegates single, pack, and all removals through `runRemoveSkill`, which is the current skill-only path in the CLI. That leaves no planned implementation for removing agent files or their synced provider views.
- Fix: Expand the remove phase to add agent-aware removal logic, either by generalizing the underlying remove command to handle both content types or by introducing a parallel agent-removal path with equivalent manifest/provider-view cleanup and tests.
- **The plan expands scope with `oat tools info` even though upstream artifacts explicitly limit this project to five subcommands** (`plan.md:20`, `plan.md:274-337`)
- Issue: Discovery, spec, and design all define the approved command surface as `install`, `update`, `remove`, `list`, and `outdated` (`discovery.md:26-28`, `discovery.md:79-80`, `spec.md:31-35`, `design.md:13-16`). Discovery also records `oat tools info <name>` as a deferred idea (`discovery.md:115-119`). Adding p01-t05 changes the goal, task count, and implementation summary without an upstream decision that re-approves the extra command.
- Fix: Remove `oat tools info` from this plan and keep it as follow-up work, or update discovery/spec/design first so the added command is intentionally in scope before implementation starts.
### Medium
- **p04-t01 targets generated asset files instead of the source files that actually define bundled agents** (`plan.md:622-651`)
- Issue: The task tells the implementer to edit `packages/cli/assets/agents/*` directly, but those files are generated by `packages/cli/scripts/bundle-assets.sh` from `.agents/agents/*`. Editing generated assets as the primary change target risks version skew and skips the source-of-truth path the build uses.
- Fix: Change the task to update `.agents/agents/oat-codebase-mapper.md` and `.agents/agents/oat-reviewer.md`, then rebuild/regenerate `packages/cli/assets` as a verification step if bundled assets must be refreshed in-repo.
